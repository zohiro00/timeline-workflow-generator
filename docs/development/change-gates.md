# 変更時のゲート

## ブランチとレビュー

- `main` への直接 push は禁止です。変更は必ず作業ブランチで行い、レビュー可能な差分として提出してください。
- 変更前に `git status --short` と現在ブランチを確認してください。`main` 上で作業している場合は、変更前に作業ブランチを作成してください。
- 作業ブランチを作る前、または既存ブランチで作業を再開する前に、必ず `git fetch origin` で `origin/main` を最新化し、その最新の `origin/main` を基準にしてください。ローカル `main` だけを基準にしてはいけません。
- 作業するチェックアウト、通常の作業ディレクトリまたは git worktree ごとに、`git fetch origin` 後、`git merge-base --is-ancestor origin/main HEAD` で現在のブランチが最新の `origin/main` を含んでいることを確認してください。含んでいない場合は、作業前または PR 更新前に `origin/main` を merge または rebase してください。
- PR 作成前には `git diff --name-status origin/main` で差分対象を確認し、依頼と無関係な削除・巻き戻し・依存更新が混ざっていないことを確認してください。上流の最新変更を消す差分が見えた場合は、先にブランチを `origin/main` へ追いつかせてから作業差分を載せ直してください。
- PR またはレビュー依頼では、変更内容、影響範囲、追加・更新したテスト、実行した検証コマンドを必ず明記してください。
- `pnpm run guard:change` で、保護ブランチ上の作業と採用外ロックファイルを検知します。
- `pnpm run install:hooks` を実行すると、`.githooks/pre-push` から `pnpm run guard:change` が呼ばれます。Git hook はローカル設定なので、CI や GitHub の branch protection と併用してください。

## 完了条件

- コード変更、仕様変更、バグ修正、リファクタリングは、対応するテスト追加または既存テスト更新とセットで行ってください。
- 変更後は `pnpm test` を必ず実行し、全テストが green になるまで完了扱いにしないでください。
- push 前に `pnpm run guard:change` を実行してください。
- UI、SVG 出力、ビルド設定、依存関係に関わる変更では `pnpm run build` も必ず実行してください。
- 型チェック用 script が追加された後は、変更後に必ずその script も実行してください。現時点の [../../package.json](../../package.json) には型チェック用 script がないため、存在しないコマンドを実行済みとして扱わないでください。
- `pnpm test`、`pnpm run build`、型チェックのいずれかが失敗した状態で、変更を完了・レビュー可能・マージ可能として扱ってはいけません。
- 検証できない項目がある場合は、未検証のまま通したり代替結果で置き換えたりせず、理由と残リスクを報告してください。

## 継続的改善ループ

- 同じ種類の失敗、やり直し、権限ミス、検証漏れ、判断迷いが累積で 3 回観測された場合は、その場の修正だけで終わらせず、再発防止の改善を行ってください。
- 回数は同一セッション内だけで数えません。archives、docs、PR コメント、作業ログ、今回の観測を含めて累積で判断します。
- archives に同種の記録がある場合は、失敗パターンIDをセッション横断の累積キーとして照合します。
- 3 回は改善開始の上限です。重大、低コスト、明らかな再発の場合は、1 回または 2 回でも改善してかまいません。
- 改善は、現行 docs への方針追加、archives への pitfall 記録、AGENTS.md から参照される細則への追加、guard script や test による自動検知、デフォルト手順の変更のいずれかに昇格してください。
- 改善後は、次回どのタイミングで検知または回避されるかを明記してください。

## 作業前後の確認

- 作業前に `git status --short` を確認し、ユーザーの未コミット変更を上書きしないでください。
- 作業前後と PR 作成前に、比較対象が最新の `origin/main` であることを確認してください。古いローカル `main` との差分確認だけで完了扱いにしないでください。
- `git merge-base --is-ancestor origin/main HEAD` は hook では強制しません。remote の最新化状態や一時的なオフライン作業に左右されやすいため、`pnpm run guard:change` には含めず、PR 作成前・PR 更新前の明示確認として扱います。
- 既存の変更があるファイルを編集する場合は、差分を読んで意図を壊さないようにしてください。
- 作業後は `git diff --check` を実行し、不要な空白や改行の問題を確認してください。
- 変更内容、実行したテスト、未確認事項を簡潔に報告してください。

## 禁止事項

- 実際の環境、実行結果、コード、設定と異なる内容をドキュメントに書かないでください。
- 実行していないコマンドを実行済みとして報告しないでください。
- `main` に直接 push しないでください。
- テストが失敗している変更、未実行の変更、テストなしの機能変更を完了扱いにしないでください。
- `pnpm` 前提のプロジェクトで、npm や別のパッケージマネージャーへのフォールバックを追加しないでください。
- `package-lock.json` など、採用していないパッケージマネージャーのロックファイルを作らないでください。

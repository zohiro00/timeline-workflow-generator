# 変更されうる値のハードコード

## 何が起きたか

SVG の配色プリセットを追加した際、濃い青の色値を複数のテーマ定義へ直接書いていました。

その後「濃い青はすぐ変えたくなるかもしれない」という観点で見直すと、色値が複数箇所へ散っており、変更時に影響範囲を検索で確認する必要がありました。

## 原因

- まず表示結果を作ることを優先し、将来変更される可能性が高い値を設計上のトークンとして扱わなかった。
- 「同じ意味の値」と「たまたま同じ文字列の値」を区別する責務境界を作らなかった。
- プリセットという変更されやすい仕様を、色トークンとテーマ定義に分けずに直接展開した。

## アンチパターン

- 色、寸法、ラベル、判定条件、プリセット値など、将来変更されうる値を複数箇所へ直接書く。
- 変更時に `rg` で値そのものを探さないと影響範囲が分からない状態にする。
- 同じ意味の値を名前付きの定数、設定オブジェクト、スキーマ、fixture に集約しない。
- テストの期待値と production code の設定値を区別せず、どちらも同じ感覚でハードコードする。

## 今後の方針

- ハードコードは基本的に NG と扱う。
- 将来変更されうる値は、責務に合う場所へ名前付きで集約する。
- 同じ意味の値は 1 箇所から参照する。例: ブランド色やテーマ色は `themeColor` のようなトークンに寄せ、テーマ定義はそのトークンを参照する。
- リテラル値を直接書いてよいのは、標準仕様値、構文上の固定文字列、局所的な計算値、テストで固定したい期待値など、変更可能性と影響範囲を説明できる場合に限る。
- テストでは出力として守るべき値を明示してよい。ただし production code 側の変更点を複数箇所へ散らす理由にはしない。

## 再発防止チェック

- 新しい値を追加したら「この値は将来変わるか」「同じ意味で再利用されるか」を確認する。
- 同じ値を 2 箇所以上に書く前に、定数・設定・スキーマ・fixture のどこへ集約すべきか判断する。
- 変更レビューでは、値そのものではなく名前で影響範囲を追えるか確認する。
- `rg` で値を検索した結果、production code に同じ意味のリテラルが散っている場合は集約する。

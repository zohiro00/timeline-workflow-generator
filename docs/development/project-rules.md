# プロジェクト構造と設計ルール

## プロジェクトの前提

- このプロジェクトは、Markdown 内の `workflow` コードブロックまたは DSL 本文から、依存関係ベースのレーン型タイムライン SVG を生成する Web アプリです。
- コア処理は [../../src/workflow.js](../../src/workflow.js) に集約し、パーサー、DAG レイアウト、SVG レンダリングを DOM に依存しない純粋な JavaScript として保ちます。
- UI は [../../src/main.js](../../src/main.js) と [../../src/styles.css](../../src/styles.css) に閉じ込め、コアロジックへブラウザ状態や DOM 操作を混ぜないでください。
- 要件の根拠は [../requirements_specification.md](../requirements_specification.md) を参照してください。

## 変更に強い設計ルール

- DSL は宣言的に保ちます。ユーザーに座標、空白、ハイフン数、描画位置の微調整を要求する仕様を追加してはいけません。
- 横位置 `gridX` は依存関係の DAG から計算します。手動指定を入れる場合も、既存の最長経路レイアウトを壊さない明確な意味とテストを先に定義してください。
- コアのデータ構造は、小さく明示的なオブジェクトに保ちます。暗黙の副作用、グローバル状態、DOM 参照、表示都合だけのフラグを `workflow.js` に持ち込まないでください。
- 単一責務を優先します。1つの関数、モジュール、ファイルに、入力取得、パース、検証、レイアウト、レンダリング、UI操作、サンプル管理など複数の変更理由を持たせないでください。
- サンプルコード、テスト fixture、UI の初期値、仕様説明は、それぞれの責務が分かる場所に置きます。重複を避けるために共有する場合も、利用側の都合を共有元へ混ぜ込まないでください。
- 変更時は「どの責務を変える差分か」を説明できる単位に分けます。責務境界をまたぐ変更では、境界を越えるデータ構造とテストを明示してください。
- パース、検証、レイアウト、レンダリングの責務を混ぜないでください。仕様変更は、どの層の変更かを分けて実装します。
- ハードコードは基本的に避けます。色、寸法、ラベル、プリセット、判定条件など将来変更されうる値は、責務に合う設定オブジェクト、定数、スキーマ、テスト fixture に集約し、影響範囲を追える名前を付けてください。
- 同じ意味の値を複数箇所へ直接書かないでください。変更時に検索しないと影響範囲が分からない実装は、修正漏れを生む変更に弱い設計として扱います。
- 外部の巨大なレイアウトエンジンには依存しません。依存を追加する場合は、可搬性、保守性、バンドルサイズ、テスト容易性への影響を説明できることが条件です。
- SVG 出力は文字列として扱うため、ユーザー入力を出力する箇所では必ず XML エスケープを維持してください。

## 実装スタイル

- JavaScript は ES modules で書きます。
- 既存の標準ライブラリ中心の実装を尊重し、不要なフレームワークやビルド設定を増やさないでください。
- リテラル値を直接書く場合は、その値が局所的で、標準仕様・構文・テスト期待値などとして意味が明確であり、将来変更時の探索範囲を広げないことを確認してください。
- 例外は `WorkflowError` を使い、ユーザーが DSL のどこを直せばよいか分かるメッセージを返してください。
- コメントは、読み手の判断を助ける複雑な処理にだけ短く追加します。コードをそのまま言い換えるコメントは不要です。
- 既存の日本語 UI とエラーメッセージのトーンを合わせます。利用者向け文言は具体的で、修正行動につながる表現にしてください。
- 変更範囲は狭く保ちます。依頼に無関係なリファクタリング、配色変更、文言変更、依存更新は混ぜないでください。

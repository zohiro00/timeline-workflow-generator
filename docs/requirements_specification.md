# 時系列ワークフロー図（仮称）ジェネレーター 要件定義書
## 〜 変更に強い依存関係ベースのレーン型タイムライン生成ツール 〜
---

## 1. はじめに

### 1.1 開発の背景
システムの業務フローや申請プロセスの設計において、「どの機能（レーン）で」「いつ（時系列）」イベントが発生するかを可視化することは極めて重要である。

既存 of テキストベース描画ツール（Mermaid.js等）は優れた表現力を持つが、複数レーンが並行し、かつ時系列の「縦のライン」を厳密に同期させるレイアウト（シーケンス図の横型配置のようなイメージ）の描画においては、トポロジカルな自動配置エンジンの制約により、意図通りのグリッド配置を維持することが困難であった。また、手動でスペースやハイフンの数を調整して位置を微調整するアプローチでは、プロセスの追加・変更時に全体のレイアウト定義が崩れるため、「変更に弱い」という致命的な課題があった。

### 1.2 開発の目的
本プロジェクトでは、ユーザーが「ノードの定義」と「ノード間の依存関係（繋がり）」のみをプレーンテキストで宣言するだけで、システム側が自動的に時間軸（グリッドX）を計算・同期し、かつ変更に対して極めて高い堅牢性を持つ、独自のSVGワークフロー図ジェネレーターを開発する。

---

## 2. 製品概要とポジショニング（棲み分け）

本システムは、既存のMermaid.jsファミリーがカバーしていない「時間同期」と「機能レーン」のクロスセクションを埋めるツールとして位置づける。

| 比較軸 | Mermaid.js (flowchart) | Mermaid.js (sequenceDiagram) | 本システム (時系列ワークフロー図) |
| :--- | :--- | :--- | :--- |
| **時間軸の方向** | 自由（自動配置に依存） | 上から下へ流れる | **左から右へ流れる（完全固定）** |
| **縦軸の役割** | 特になし（混在） | ライフライン（オブジェクト） | **レーン（機能・申請の種類）** |
| **時間同期（縦の配置）** | 揃わない（ガタガタになる） | メッセージ単位で揃う | **グリッド（ステップ数）で完璧に揃う** |
| **変更への強さ** | 追加により全体配置が激変する | 強い | **強い（トポロジーを崩さず自動追従）** |

## 3. ユーザー表記方法（Markdownワークフロー記法）仕様

ユーザーはMarkdownのコードブロック内に、レーン、ノード、依存関係をセクション単位で記述する。座標や余白に依存せず、完全な宣言型で記述する。

### 3.1 記述例

```workflow
# 申請ワークフローの時系列図

## lanes
- a: a申請
- b: b申請
- c: c申請

## nodes
- a
  - a1: 作成
  - a2: 承認
  - a3: 保留
  - a4: 取消
- b
  - b1: 作成
  - b2: 承認

## workflow
- a1 -> a2
- a2 -> b1     %% a申請の「承認」完了後、b申請の「作成」へ移行
- b1 -> b2
- b1 -.-> a4   %% bの作成中に、aを「取り消す」というレーン間の交差を定義
- a2 -> a3 -> a4
```

## 4. 機能要件とアルゴリズム

### 4.1 パーサー（解析エンジン）機能
- **ブロック抽出機能**: Markdownドキュメント内から ` ```workflow ` で囲まれたテキスト領域を正規表現等で抽出する。
- **行解析機能**: Markdown見出し（`#`, `## lanes`, `## nodes`, `## workflow`）、Markdownリスト、矢印（`->`, `-.->`）を1行ずつパースし、エラーがある場合は適切な行番号とメッセージを返却する。

### 4.2 レイアウトエンジン（時系列位置の自動計算アルゴリズム）
「変更に強い」構造を実現するため、ノードの横位置（グリッドX座標）は、ユーザーの指定ではなく、依存関係の有向グラフ（DAG）から以下のアルゴリズム（最長経路法）を用いて算出する。

> 💡 **グリッドX（時系列ステップ数）の算出ルール**
> 1. いかなるノードからも矢印を向けられていないルートノード（例: `a1`）の `gridX` を `0` とする。
> 2. 依存関係 `From -> To` が存在する時、`To` のノードの `gridX` は `(FromのgridX + 1)` とする。
> 3. 単一のノードに複数の依存ルートが合流・交差する場合（例: `a4`）、該当ノードの `gridX` は **`max(すべてのFromのgridX) + 1`** とする。

### 4.3 レンダラー（SVG描画エンジン）機能
- **絶対位置グリッドレンダリング**: 各ノードのX座標は `paddingLeft + (gridX * gridXSize)`、Y座標は `paddingTop + (laneIdx * gridYSize)` で完全固定算出する。
- **スマートコネクタ（エッジ描画）**:
  - 同レーン内の右方向への遷移は、実線の直線（`->`）または点線の直線（`-.->`）で描画する。
  - 他レーンへの遷移や、時系列を跨ぐ交差、または逆行（トリガー等）が発生する場合、直線ではなく、ノードを綺麗に回避する**3次ベジェ曲線（Cubic Bezier）**を用いてなめらかなS字・クランク線を描画する。

## 5. 非機能要件・環境要件

### 5.1 開発言語および実行環境
- **言語**: Node.js (JavaScript / TypeScript) または Python 3.x。環境に応じて移植可能な設計とする。
- **依存ライブラリ**: 外部の巨大なレイアウトエンジン（Graphviz等）には依存せず、軽量なテキスト処理、標準のファイルI/O、および標準のSVG文字列組み立てのみで完結させる（可搬性の最大化）。

### 5.2 リアルタイムプレビュー（開発環境）
- **Viteを用いたHMR（Hot Module Replacement）環境**: フロントエンドツール「Vite」を活用し、ユーザーがDSLテキスト（Markdownファイル）を書き換えて保存した瞬間に、ブラウザ側の表示をミリ秒単位で自動追従・更新するプレビュー機能を提供する。
